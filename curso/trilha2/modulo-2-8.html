Composicao de Componentes

O que e

Composicao e o padrao de construir componentes complexos a partir de componentes menores. Um Card compoe Header, Body e Footer. Uma Table compoe Header, Row, Cell. Permite reutilizacao e flexibilidade.

Por que aprender

Dashboards tem muita repeticao visual com variacoes sutis. Composicao permite criar variantes sem duplicar codigo. Um Card de metrica e um Card de grafico usam o mesmo Card base.

Conceitos-chave

// Composicao com compound components
// Card.js
const Card = ({ children, className }) => (
  <div className={`bg-dark-800 rounded-xl border border-dark-600
                   ${className}`}>
    {children}
  </div>
);

Card.Header = ({ children }) => (
  <div className="px-6 py-4 border-b border-dark-600">{children}</div>
);

Card.Body = ({ children }) => (
  <div className="p-6">{children}</div>
);

Card.Footer = ({ children }) => (
  <div className="px-6 py-4 border-t border-dark-600 bg-dark-700/30">
    {children}
  </div>
);

// Uso - MUITO flexivel
<Card>
  <Card.Header>
    <h3>Titulo do Card</h3>
  </Card.Header>
  <Card.Body>
    <p>Conteudo principal aqui</p>
    <Chart data={data} />
  </Card.Body>
  <Card.Footer>
    <Button>Ver mais</Button>
  </Card.Footer>
</Card>

// Card simples (sem header/footer)
<Card>
  <Card.Body>Apenas conteudo</Card.Body>
</Card>

Props e Variantes

O que e

Props sao parametros passados para componentes. Variantes sao conjuntos pre-definidos de props para casos comuns. Um Button pode ter variantes "primary", "secondary", "danger" que definem cores e estilos.

Por que aprender

Props permitem customizacao. Variantes garantem consistencia. Sem variantes, cada uso do Button pode ter cores ligeiramente diferentes. Variantes enfor√ßam design system.

Conceitos-chave

// Sistema de variantes com Tailwind
const buttonVariants = {
  variant: {
    primary: 'bg-blue-600 hover:bg-blue-500 text-white',
    secondary: 'bg-dark-700 hover:bg-dark-600 text-neutral-200',
    danger: 'bg-red-600 hover:bg-red-500 text-white',
    ghost: 'bg-transparent hover:bg-dark-700 text-neutral-300',
  },
  size: {
    sm: 'px-3 py-1.5 text-sm',
    md: 'px-4 py-2',
    lg: 'px-6 py-3 text-lg',
  }
};

function Button({
  variant = 'primary',
  size = 'md',
  children,
  ...props
}) {
  const classes = `
    rounded-lg font-medium transition-colors
    ${buttonVariants.variant[variant]}
    ${buttonVariants.size[size]}
  `;

  return <button className={classes} {...props}>{children}</button>;
}

// Uso
<Button>Primary (default)</Button>
<Button variant="secondary" size="sm">Secondary Small</Button>
<Button variant="danger" size="lg">Delete</Button>

// Com class-variance-authority (CVA) - mais robusto
import { cva } from 'class-variance-authority';

const button = cva('rounded-lg font-medium transition-colors', {
  variants: {
    variant: { primary: '...', secondary: '...' },
    size: { sm: '...', md: '...' }
  },
  defaultVariants: { variant: 'primary', size: 'md' }
});

Comunicacao Entre Componentes

O que e

Como componentes trocam informacao. Props fluem de pai para filho. Callbacks (onChange, onClick) fluem de filho para pai. Context ou state managers permitem comunicacao entre irmaos ou distantes.

Por que aprender

Dashboards tem muita interacao entre partes. Filtro muda ‚Üí tabela atualiza. Clique em grafico ‚Üí mostra detalhes. Sem padrao de comunicacao, codigo vira espaguete.

Conceitos-chave

// 1. PROPS DOWN (pai ‚Üí filho)
function Dashboard() {
  const [filters, setFilters] = useState({});

  return (
    <div>
      <FilterBar filters={filters} />  {/* passa dados */}
      <DataTable filters={filters} />  {/* mesmo dados */}
    </div>
  );
}

// 2. CALLBACKS UP (filho ‚Üí pai)
function FilterBar({ filters, onFilterChange }) {
  return (
    <select onChange={(e) => onFilterChange({
      ...filters,
      status: e.target.value
    })}>
      ...
    </select>
  );
}

// Uso no pai
<FilterBar
  filters={filters}
  onFilterChange={(newFilters) => setFilters(newFilters)}
/>

// 3. CONTEXT (qualquer nivel)
const FilterContext = createContext();

function Dashboard() {
  const [filters, setFilters] = useState({});

  return (
    <FilterContext.Provider value={{ filters, setFilters }}>
      <Sidebar>
        <FilterPanel />  {/* pode acessar context */}
      </Sidebar>
      <MainContent>
        <DataTable />    {/* pode acessar context */}
      </MainContent>
    </FilterContext.Provider>
  );
}

// Em qualquer componente filho
const { filters } = useContext(FilterContext);

Slots e Children

O que e

Slots sao "buracos" em componentes onde conteudo custom pode ser inserido. Em React, children e o slot padrao. Named slots permitem multiplas areas de insercao: header, content, footer.

Por que aprender

Slots permitem criar layouts flexiveis. Um Modal pode ter slot para titulo, conteudo e botoes. Usuario define o que vai em cada area, componente controla layout e comportamento.

Conceitos-chave

// Children como slot padrao
function Modal({ children, isOpen }) {
  if (!isOpen) return null;

  return (
    <div className="fixed inset-0 bg-black/50 flex items-center justify-center">
      <div className="bg-dark-800 rounded-xl p-6">
        {children}  {/* qualquer conteudo */}
      </div>
    </div>
  );
}

// Named slots com props
function Modal({ title, children, footer, isOpen }) {
  return (
    <div className="...">
      <div className="...">
        {/* Slot: titulo */}
        <div className="border-b border-dark-600 pb-4 mb-4">
          {title}
        </div>

        {/* Slot: conteudo */}
        <div className="mb-4">
          {children}
        </div>

        {/* Slot: footer */}
        <div className="border-t border-dark-600 pt-4">
          {footer}
        </div>
      </div>
    </div>
  );
}

// Uso com named slots
<Modal
  isOpen={true}
  title={<h2 className="text-xl font-bold">Confirmacao</h2>}
  footer={
    <div className="flex justify-end space-x-2">
      <Button variant="secondary">Cancelar</Button>
      <Button>Confirmar</Button>
    </div>
  }
>
  <p>Tem certeza que deseja continuar?</p>
</Modal>

Render Props e HOCs

O que e

Render Props passam funcao como children, permitindo que componente pai controle como filho renderiza dados. HOCs (Higher Order Components) envolvem componentes adicionando funcionalidade. Patterns avan√ßos para reutilizacao de logica.

Por que aprender

Sao patterns classicos de React. Embora hooks tenham substituido muitos casos, ainda sao uteis para bibliotecas de componentes e integracao com libs externas. Entender ajuda a usar libs como React Query, Formik.

Conceitos-chave

// RENDER PROPS: componente fornece dados, pai decide como renderizar
function DataFetcher({ url, children }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch(url)
      .then(res => res.json())
      .then(data => { setData(data); setLoading(false); });
  }, [url]);

  // Passa dados para funcao children
  return children({ data, loading });
}

// Uso - flexibilidade total na renderizacao
<DataFetcher url="/api/users">
  {({ data, loading }) => (
    loading
      ? <Skeleton />
      : <UserTable users={data} />
  )}
</DataFetcher>

// HOC: funcao que envolve componente
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const { user } = useAuth();

    if (!user) return <LoginRedirect />;

    return <WrappedComponent {...props} user={user} />;
  };
}

// Uso
const ProtectedDashboard = withAuth(Dashboard);

// Hoje, custom hooks sao mais comuns
function useDataFetcher(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  // ...
  return { data, loading };
}

Patterns de Layout para Dashboards

O que e

Patterns de layout especificos para dashboards: Sidebar + Main, Header + Sidebar + Main, Split View, Master-Detail. Cada pattern otimizado para diferentes tipos de dashboard.

Por que aprender

Nao reinventar a roda. Patterns testados resolvem problemas comuns de navegacao, hierarquia e uso de espaco. Escolher o pattern certo acelera desenvolvimento e melhora UX.

Conceitos-chave

<!-- PATTERN 1: Sidebar + Main (mais comum) -->
<div class="flex min-h-screen">
  <aside class="w-64 bg-dark-800 border-r border-dark-600 shrink-0">
    <nav>...</nav>
  </aside>
  <main class="flex-1 p-6">
    <!-- Conteudo -->
  </main>
</div>

<!-- PATTERN 2: Header + Sidebar + Main -->
<div class="flex flex-col min-h-screen">
  <header class="h-14 bg-dark-800 border-b">...</header>
  <div class="flex flex-1">
    <aside class="w-64">...</aside>
    <main class="flex-1">...</main>
  </div>
</div>

<!-- PATTERN 3: Master-Detail -->
<div class="flex min-h-screen">
  <!-- Master: lista -->
  <div class="w-80 border-r overflow-y-auto">
    <ul>...items...</ul>
  </div>
  <!-- Detail: conteudo do item selecionado -->
  <div class="flex-1 p-6">
    <!-- Detalhes do item -->
  </div>
</div>

<!-- PATTERN 4: Split View (comparacao) -->
<div class="flex min-h-screen">
  <div class="w-1/2 p-4 border-r">
    <!-- Painel A -->
  </div>
  <div class="w-1/2 p-4">
    <!-- Painel B -->
  </div>
</div>
